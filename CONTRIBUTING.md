# Contributing to `actinf_policy_entanglement_lean`

> **Public repository:** <https://github.com/docxology/policy_entanglement>  
> Clone with `git clone https://github.com/docxology/policy_entanglement.git`  
> Discussions: <https://github.com/docxology/policy_entanglement/discussions>

Welcome. This guide distills the rules in [`AGENTS.md`](AGENTS.md) into a
concrete onboarding flow for contributors. Read it once, bookmark
[`docs/guides/styleguide.md`](docs/guides/styleguide.md) and
[`docs/reference/lean_reference.md`](docs/reference/lean_reference.md), and
keep [`AGENTS.md`](AGENTS.md) close while you work — it is the canonical
reference if anything in this guide drifts.

## TL;DR (5-minute setup)

```bash
# 1. Sync deps (sim + viz groups recommended)
uv sync --group sim --group viz

# 2. Verify the Lean budget is 0 sorry / 0 axiom / 0 unsafe
uv run python scripts/build_lean.py

# 3. Run the full Python suite with coverage gate
uv run pytest tests/ --cov=src --cov-fail-under=95

# 4. End-to-end pipeline (figures → sim → manuscript validation)
uv run python scripts/run_all.py
```

If all four steps exit 0 you are ready to contribute.

## The eleven hard constraints

These are non-negotiable. CI enforces every one of them.

1. **Mathlib-free at the Lean boundary.** Do not add `import Mathlib...` to any module under `lean/ActinfPolicyEntanglement/` or `lean/FepSketches/`.
2. **Hygiene budget = 0.** No `sorry`, no `axiom`, no `unsafe`/`partial`/`noncomputable`. `scripts/build_lean.py` enforces this.
3. **Reserved tokens.** `Π` and `λ` are Lean 4 binder tokens; never use them as ordinary identifiers — use `Pol`, `q_lam`, `lam` etc.
4. **Namespaced Python imports.** Every test, script, and intra-package reference uses `from lean.coupling import …`, `from simulation.specs import …`, `from manuscript.registry import …`, `from visualizations.heatmaps import …`. Never bare imports; never relative imports outside the same subpackage.
5. **No mocks in tests.** Use real `numpy` arrays; if you need randomness, use `np.random.default_rng(seed=…)` with a fixed seed.
6. **Coverage floor: 95%.** CI runs `pytest tests/ --cov=src --cov-fail-under=95`.
7. **Thin orchestrators only in `scripts/`.** Every numerical computation must come from `src/`; `scripts/` handles I/O, matplotlib, and stdout-path emission.
8. **Print output paths to stdout** in scripts, one per line, so the pipeline manifest can collect them.
9. **Headless matplotlib.** Always set `MPLBACKEND=Agg` before importing `matplotlib.pyplot`; the scripts already do this.
10. **Determinism.** All scripts must run reproducibly (fixed seeds, no time-dependent state).
11. **Disposable `output/`.** Treat the entire `output/` tree as regenerable; never hand-edit it, never commit transient outputs.

## Common contribution patterns

### Adding a new Lean theorem

1. Pick the file in `lean/ActinfPolicyEntanglement/` whose namespace matches your concept (`JointDist`, `Coupling`, `FreeEnergy`, …). If no fit, ask before adding a new module.
2. State the theorem in the **boundary form** — if it requires Mathlib content (KL chain rule, SVD, Bregman Taylor), make it **witness-consuming**: take a `structure` parameter that packages the analytic witness, and certify it. See [`docs/reference/lean_reference.md`](docs/reference/lean_reference.md) for the witness-contract pattern.
3. Prove it without `sorry` / `axiom` / `unsafe`.
4. `cd lean && lake build` — must succeed.
5. `uv run python scripts/build_lean.py` — must report `0 sorry, 0 axiom, 0 unsafe`.
6. Update [`docs/reference/lean_reference.md`](docs/reference/lean_reference.md) per-file table to add a row.
7. If the theorem has a numerical companion, add it under `src/lean/` with a matching test.

### Adding an analytical helper (Python mirror)

1. Implement in `src/lean/<module>.py` with type hints on the public API.
2. Add a test in `tests/test_<module>.py`. Target ≥ 95% coverage on the new code; aim for 100% on pure-compute helpers.
3. Use `np.random.default_rng(seed=...)` if randomness is needed; never `np.random.seed`.
4. If the helper supports a manuscript claim, mirror it in the relevant Lean module so the cross-track invariant table in [`AGENTS.md`](AGENTS.md) stays current.

### Adding a manuscript section, paragraph, or claim

1. Edit the relevant `manuscript/*.md` file. **No hardcoded numbers** — every grid count, seed, λ value, T-step must come from a `[[VAR:key]]` token. See [`docs/guides/styleguide/manuscript-variables.md`](docs/guides/styleguide/manuscript-variables.md).
2. If you mention a new equation, register it in `manuscript/refs/labels.yaml` and place it with `[[EQ:label]]`. The renderer auto-numbers it `S.K`.
3. If you mention a new theorem, register it in `labels.yaml` under `theorems:` and reference it with `[[THMREF:key]]`.
4. If you cite a paper, register it in `manuscript/refs/citations.yaml` and reference it with `[[CITATION:key]]`.
5. `uv run python scripts/inject_manuscript_variables.py` — substitutes `[[VAR:...]]` tokens.
6. `uv run python scripts/validate_manuscript.py` — fails if any token unresolved, any citation missing, any hardcoded numeric literal slipped in.

### Adding a figure

1. Implement the figure-generating function in `scripts/generate_figures.py` or `scripts/simulate_pymdp.py`. The function must accept no required args and **return the output path**.
2. Use a helper from `src/visualizations/` for the actual plotting — never re-implement plotting logic in scripts.
3. Tag every PNG with `figure_metadata(...)` (script, function, hyperparameter snapshot, git revision, ISO timestamp) so the audit page (`docs/reference/veridical_status.md`) can verify provenance.
4. Add the figure to `manuscript/refs/labels.yaml` under `figures:` so it gets numbered, and reference it with `[[FIG:label]]` in the prose.
5. Print the path to stdout (one line, one path).

### Adding a test

1. Place it in `tests/test_<module>.py` mirroring the source location.
2. **Never use `MagicMock`, `mocker.patch`, or any mocking framework.** Use real arrays, real PMFs, real files (`tmp_path` fixture for I/O).
3. Use `pytest.mark.requires_pymdp` to skip when `pymdp` is missing; use `pytest.mark.slow` for tests that take longer than ~1 s.
4. The test must run deterministically. If randomness is needed, fix the RNG seed.
5. `uv run pytest tests/test_<module>.py -v` — must pass.

## Development environment

* **Python:** 3.10–3.12 (CI matrix tests all three).
* **Lean:** 4.29.0, pinned via `lean/lean-toolchain`.
* **Package manager:** `uv` (`uv sync --group sim --group viz`).
* **IDE:** point your interpreter at `.venv/bin/python` after `uv sync`.

## Code review expectations

* **Lean:** every new theorem has a row in `docs/reference/lean_reference.md`. Every new module has the `Mathlib`-free header comment.
* **Python:** every new public function has type hints, a docstring, and at least one direct unit test.
* **Manuscript:** every new claim has a citation or a Lean / Python pointer; every new number is a `[[VAR:...]]`.
* **Coverage:** PRs that drop coverage below 95% are blocked by CI; aim to leave coverage ≥ where it started.

Welcome — the bar is high, but the structure is forgiving once you see it. If a constraint feels arbitrary, read `AGENTS.md`'s "Repository Constitution" section before working around it; every rule is there for a reason.
